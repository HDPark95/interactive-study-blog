# 기술 문서 작성 스타일 가이드

> 이 가이드는 토스 테크니컬 라이팅 원칙을 기반으로 합니다.

## 핵심 원칙

### 1. 문서 유형 정하기

| 유형 | 목적 | 필수 요소 |
|------|------|-----------|
| 학습 문서 | 새로운 개념 습득 | 학습 목표, 사전 준비, 단계별 안내, 실행 가능한 예제 |
| 설명 문서 | 깊은 이해 | 등장 배경, 기본 원리, 다른 접근법 비교, 시각 자료 |
| 문제 해결 | 트러블슈팅 | 문제 정의, 원인 설명, 단계별 해결법 |
| 참조 문서 | 빠른 확인 | 간결한 개요, 파라미터, 반환값, 예제 |

### 2. 정보 구조 원칙

1. **한 페이지 하나 목표**: H4 이상 제목이 필요하면 페이지 분리 고려
2. **개요 필수**: 문서 시작에 "이 문서에서 얻을 것"을 명확히
3. **예측 가능한 구조**: 일관된 제목 패턴, 기본→상세 순서
4. **가치 우선**: 기능 설명보다 "왜 필요한가"를 먼저
5. **효과적 제목**: 핵심 키워드 포함, 30자 이내

### 3. 문장 다듬기

1. **간결성**: 한 문장에 하나의 생각
2. **메타 담화 최소화**: "앞서 설명했듯이" 같은 표현 제거
3. **구체성**: "빠르다" → "응답 시간 50ms 이내"
4. **일관성**: 동일 개념은 동일 용어로
5. **명확한 주체**: 능동형 문장 사용

## 교육용 문서 작성 원칙 (시니어 → 주니어)

### 왜 이걸 알아야 하는가? (동기 부여)

```
Bad:  "Thread는 프로세스 내의 실행 단위입니다."
Good: "서버에 100명이 동시 접속하면 어떻게 처리할까요? Thread를 이해하면 답을 알 수 있습니다."
```

### 실무에서 언제 만나는가? (맥락 제공)

```
Bad:  "데드락은 두 스레드가 서로 락을 기다리는 상황입니다."
Good: "결제 API가 갑자기 멈췄습니다. 로그를 보니 'waiting for lock'만 반복됩니다.
       이게 바로 데드락입니다. 실제로 연 1-2회는 겪게 됩니다."
```

### 직접 경험하게 하라 (체험 학습)

```
Bad:  도표와 설명으로만 개념 전달
Good: 버튼 하나로 데드락 발생 → 시각적으로 스레드 상태 확인 → 해결 과정 체험
```

### 점진적 복잡도 (스캐폴딩)

```
Level 1: 가장 단순한 케이스 (Thread 1개)
Level 2: 약간의 복잡도 추가 (Thread 2개, 공유 자원)
Level 3: 실무 시나리오 (Thread Pool, 동시 요청)
```

### 안티패턴과 베스트 프랙티스

```
이렇게 하면 망합니다: synchronized 남발 → 성능 저하 실제 측정
이렇게 하세요: 적절한 락 범위 → 개선된 성능 비교
```

## 문서 템플릿

```tsx
// 페이지 구조
<article>
  {/* 1. 브레드크럼 */}

  {/* 2. 헤더: 제목 + 한 줄 요약 */}

  {/* 3. 왜 알아야 하나? (동기 부여) */}
  <section>
    <h2>왜 알아야 하나요?</h2>
    {/* 실무 상황 예시로 시작 */}
  </section>

  {/* 4. 한눈에 보기 (핵심 요약) */}
  <section>
    <h2>한눈에 보기</h2>
    {/* 표 또는 다이어그램으로 전체 그림 */}
  </section>

  {/* 5. 핵심 개념 (점진적 설명) */}
  <section>
    <h2>핵심 개념</h2>
    {/* 기본 → 심화 순서로 카드 배치 */}
  </section>

  {/* 6. 실무 패턴 */}
  <section>
    <h2>실무에서는</h2>
    {/* 베스트 프랙티스 + 안티패턴 */}
  </section>

  {/* 7. 직접 체험하기 */}
  <section>
    <h2>직접 체험하기</h2>
    {/* 인터랙티브 시뮬레이터 */}
  </section>

  {/* 8. 더 알아보기 (선택) */}
  <section>
    <h2>더 알아보기</h2>
    {/* 심화 주제 링크 */}
  </section>
</article>
```

## 체크리스트

### 작성 전
- [ ] 이 문서의 목표가 명확한가?
- [ ] 대상 독자(주니어)의 수준을 고려했는가?
- [ ] "왜"에 대한 답이 준비되어 있는가?

### 작성 중
- [ ] 한 섹션에 하나의 개념만 다루는가?
- [ ] 실무 예시가 포함되어 있는가?
- [ ] 시각 자료(표, 다이어그램)를 활용했는가?
- [ ] 직접 체험할 수 있는 요소가 있는가?

### 작성 후
- [ ] 전문 용어를 처음 사용할 때 설명했는가?
- [ ] 문장이 간결한가? (한 문장 = 한 생각)
- [ ] 메타 담화("앞서 말했듯이")를 제거했는가?
- [ ] 능동형 문장인가?
